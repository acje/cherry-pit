# Distributed Systems ADR Review — Most Impactful Improvements

Date: 2026-04-29

Perspective: professor of distributed systems design, focusing on correctness under partial failure, explicit consistency semantics, fault tolerance, interoperability, security, and operational recovery.

Corpus checked with `cargo run -p adr-fmt -- --tree` and `cargo run -p adr-fmt -- --lint`: 149 ADRs, 0 lint errors, 22 warnings. The warnings are tier/layer tension warnings concentrated in governance ADRs, especially S-tier `adr-fmt` decisions with B/C-tier enforcement rules.

## Corpus-level highest-impact improvements

1. **Define one end-to-end failure model.** The corpus should name the expected semantics for crash, retry, cancellation, timeout, duplicate delivery, replay, clock skew, storage corruption, split brain, and recovery.
2. **Mechanize distributed invariants.** Single-writer ownership, idempotency, sequence continuity, schema compatibility, and crash safety need property tests, fault-injection tests, deterministic simulation, or conformance vectors.
3. **Make compatibility explicit for persisted and wire-visible surfaces.** Schema evolution, rolling upgrades, rollback, golden fixtures, and migration state machines should be specified wherever bytes or events outlive a process.
4. **Strengthen ownership fencing and recovery.** Single-writer assumptions are excellent, but distributed correctness requires lease/epoch fencing, stale-writer rejection, startup reconciliation, and split-brain handling.
5. **Treat observability and security as operational protocols.** Telemetry cardinality, redaction, certificate/key rotation, tamper evidence, supply-chain provenance, and resource-limit monitoring should be mandatory controls.

## Common Domain (COM)

- **COM-0001. Complexity Budget — Strategic Investment**: Add an explicit distributed-interaction complexity budget for coordination, retries, partial failure, and rollout coupling. Local module simplicity can hide global complexity; distributed correctness failures usually emerge at interaction boundaries.
- **COM-0002. Deep Modules Over Shallow Abstractions**: Require every deep module to expose its failure, ordering, idempotency, and consistency contract. Hiding implementation complexity is good, but hiding distributed semantics forces callers to reason from guesses.
- **COM-0003. Pull Complexity Downward**: Add retry bounds, deadlines, and escalation rules for errors absorbed internally. Masked failures without bounded recovery can become silent data loss, retry storms, or availability collapse.
- **COM-0004. Different Layer, Different Abstraction**: Require each layer boundary to state whether it is also a transaction, consistency, or trust boundary. Distributed systems fail when semantic layering and atomicity/security boundaries are accidentally misaligned.
- **COM-0005. Define Errors Out of Existence**: Refine “unrecoverable conditions use panic” into a crash-containment policy. Panics may be correct for invariant violation, but uncontrolled process failure can harm availability and recovery correctness.
- **COM-0006. Interface Documentation Before Implementation**: Make failure modes, ordering guarantees, idempotency, and cancellation semantics mandatory in public interface docs. Callers across process or async boundaries cannot infer these properties from implementation.
- **COM-0007. Information Hiding — Minimize Leakage**: Add versioned ownership for hidden protocols, file formats, and schemas with compatibility tests. Hiding representation reduces coupling only if persisted and wire-visible decisions remain safely evolvable.
- **COM-0008. Design It Twice**: Require alternatives to be evaluated under crash, retry, partition, and duplicate-delivery scenarios. Designs that look equivalent locally often diverge sharply under distributed failure.
- **COM-0009. Consistency as Complexity Reducer**: Define where semantic inconsistency is allowed for compatibility or protocol evolution. Strict uniformity can conflict with backward compatibility, staged migrations, and external protocol constraints.
- **COM-0010. Code Should Be Obvious**: Extend obviousness to concurrency and temporal behavior, not just local readability. Distributed bugs often arise from hidden ordering dependencies that are syntactically clear but temporally non-obvious.
- **COM-0011. Trade-Off Analysis as First-Class Practice**: Require CAP/PACELC-style trade-off statements for persistence, messaging, and coordination decisions. Distributed architecture must explicitly choose consistency, availability, latency, and partition behavior.
- **COM-0012. Dependency Rule — Inward-Only Knowledge Flow**: Add an anti-corruption boundary rule for external protocols and generated types. Inward dependencies protect source structure, but protocol semantics can still leak into domain models.
- **COM-0013. Evolutionary Design Over Predictive Architecture**: Define compatibility-preserving evolution mechanisms for deferred distributed features. Once data or messages are persisted, “defer until needed” must still preserve future migration paths.
- **COM-0014. Modular Boundaries Aligned to Rate of Change**: Add rollout and deployment coupling as a boundary criterion. Independently changing components must also tolerate independent deployment and version skew.
- **COM-0015. Communicate Architecture Through Structure**: Include runtime topology and failure-domain structure, not only source/dependency structure. Distributed correctness depends on placement, ownership, queues, stores, and trust boundaries that source layout alone cannot show.
- **COM-0016. Dependencies as Managed Liabilities**: Add provenance and maintainer-risk scoring for dependencies, not just count/features/licenses. Supply-chain compromise is a correctness and security risk even when APIs are stable and licenses are acceptable.
- **COM-0017. Mechanized Invariant Enforcement**: Require distributed invariants to name their strongest feasible verification method. Ordering, fencing, idempotency, and crash safety need property, model, or fault-injection tests beyond linting.
- **COM-0018. Single-Writer Concurrency as Foundational Invariant** *(Proposed)*: Define failover, lease/fencing, and split-brain behavior for single-writer ownership. Single-writer designs are only safe if ownership transfer is linearizable or fenced under failure.
- **COM-0019. Design for Observability at Every Abstraction Layer** *(Proposed)*: Add required cardinality limits, sampling policy, and redaction rules for telemetry. Observability can itself become a denial-of-service vector or confidentiality leak.
- **COM-0020. Make the Safe Path the Only Path**: Define controlled escape-hatch governance for migrations, repair tools, and recovery operations. Real systems need exceptional maintenance paths; making them ad hoc is less safe than explicitly constrained authority.
- **COM-0021. Design Public Interfaces for Additive Evolution**: Add explicit compatibility matrices for wire, persisted, and public Rust API evolution. Different compatibility surfaces fail differently, especially under rolling upgrades and replay.
- **COM-0022. Pre-validate Then Commit**: Extend the rule to multi-resource commits with outbox, idempotency, or compensating-action guidance. Local atomicity does not imply distributed atomicity across storage, messaging, and side effects.
- **COM-0023. Checked Arithmetic — Explicit Overflow Handling**: Add monotonicity and wraparound tests for sequence numbers under long-running operation. Distributed logs rely on total ordering; overflow or reset can corrupt replay and fencing assumptions.
- **COM-0024. Testing Strategy Aligned to Architectural Layers**: Add fault-injection and deterministic simulation for crash/retry/reordering scenarios. Conventional unit/integration tests rarely expose distributed interleavings and recovery bugs.

## Cherry Domain (CHE)

- **CHE-0001. Design Priority Ordering**: Add a conflict-resolution matrix for distributed failure cases. Priority ordering is only useful if it deterministically resolves tensions like availability versus consistency, replay safety versus latency, and operational recovery versus type purity.
- **CHE-0002. Make Illegal States Unrepresentable**: Extend the decision with mandatory boundary-validation rules for deserialization and migration. Event-sourced systems repeatedly rehydrate from persisted bytes where type-level guarantees must be re-established at the log boundary.
- **CHE-0003. Compile-Time Error Preference**: Add a taxonomy of invariants that cannot be compile-time checked and therefore require runtime protocols. Concurrency conflicts, duplicate delivery, clock skew, and storage corruption are distributed correctness properties, not type-system properties.
- **CHE-0004. Event-Driven Architecture with DDD and Hexagonal Architecture**: Specify the consistency model for reads, projections, and catch-up. Event sourcing needs explicit visibility and replay guarantees or users will infer stronger read-after-write semantics than the architecture provides.
- **CHE-0005. Single-Aggregate Design with Compile-Time Type Safety**: Define the canonical cross-aggregate coordination pattern and its failure semantics. Single-aggregate safety is strong locally, but distributed workflows need explicit saga, compensation, and duplicate-command rules.
- **CHE-0006. Single-Writer Assumption Per Aggregate**: Add lease or epoch fencing for ownership transfer and crash recovery. “One OS process” is not a distributed invariant unless routing, stale writers, and split-brain recovery are specified.
- **CHE-0007. Forbid Unsafe Code**: Add dependency unsafe-audit requirements. Memory-safety and correctness risk still enter through dependencies that perform low-level synchronization, serialization, and file I/O.
- **CHE-0008. Pure Command Handling**: Define deterministic-input rules forbidding hidden time, randomness, and global state in handlers. Replayability and event-sourcing correctness require identical command/state inputs to produce identical events.
- **CHE-0009. Infallible apply() on Aggregate and Projection**: Define corruption containment instead of process-wide panic as the only operational response. A single bad event should poison or quarantine a stream, not necessarily crash unrelated aggregates or projections.
- **CHE-0010. DomainEvent Supertrait Bounds**: Add a stable event-discriminator registry rule. Immutable `event_type()` strings need collision detection and ownership, especially across crates and bounded contexts.
- **CHE-0011. AggregateId as NonZeroU64 Newtype**: Add aggregate-type or context namespace semantics to IDs. `NonZeroU64` is safe locally but not globally meaningful without preventing cross-context ID confusion in messages and references.
- **CHE-0012. Aggregate Default for Zero-State Construction**: Model “uninitialized” explicitly in guidance or test contracts. A default-only state is not a valid aggregate, and replay/creation correctness depends on never dispatching against it accidentally.
- **CHE-0013. Create/Send Split in CommandGateway and EventStore**: Add idempotent create semantics. Externally retried create commands can allocate multiple aggregate IDs unless creation has a request key or domain uniqueness protocol.
- **CHE-0014. Commands Not Serializable by Default**: Specify the boundary contract for remote command ingress. Once commands cross process boundaries, validation, authentication, idempotency, and schema evolution become correctness concerns.
- **CHE-0015. Error Type Per Command, Not Per Aggregate**: Add a classification layer for retryable versus terminal errors. Distributed dispatch needs machine-readable failure semantics for retries, compensation, and dead-letter handling.
- **CHE-0016. Store-Created Envelopes with Correlation and Causation Tracking**: Require envelope metadata invariants across batches. Sequence, timestamp, correlation, and causation must form a coherent causal record for replay debugging and deduplication.
- **CHE-0017. Policy Output as Static Associated Type**: Define policy-output dispatch guarantees. Static typing prevents wrong output shapes, but correctness also needs ordered dispatch, retry behavior, and failed-output handling.
- **CHE-0018. Synchronous Domain Logic, Asynchronous Infrastructure**: Add cancellation and timeout semantics at the async boundary. Infrastructure futures can be cancelled or time out after partial effects, affecting at-least-once behavior and retries.
- **CHE-0019. Load Returns Empty Vec, Not NotFound Error**: Distinguish “missing stream” from “empty but valid stream” at the storage metadata level. Treating absence as empty is elegant, but operational recovery and append validation need precise stream-existence semantics.
- **CHE-0020. Infrastructure-Owned Aggregate Identity**: Add domain uniqueness guidance for natural keys. Store-assigned IDs prevent allocation conflicts but do not prevent duplicate domain entities under retried or concurrent create requests.
- **CHE-0021. Non-Exhaustive Error Types for Semver Safety**: Add stable error-code categories. Wildcard matching preserves semver, but distributed clients need durable categories for retry, compensation, and alerting.
- **CHE-0022. Event Schema Evolution Strategy**: Add explicit upcaster and rollback policy. Additive-only evolution still fails under mixed-version deployments and rollback after new variants are written.
- **CHE-0023. Aggregate Lifecycle States**: Add recommended lifecycle invariant tests. Lifecycle is domain-owned, but post-termination command rejection is a common correctness invariant that should be systematically verified.
- **CHE-0024. Event Delivery Model**: Add a durable outbox or checkpointed catch-up protocol. Persist-then-publish with best-effort notification is incomplete without a specified way for consumers to discover missed events.
- **CHE-0025. RPITIT Over async_trait**: Add an object-safety escape-hatch decision for integration boundaries. Monomorphized ports are efficient, but distributed adapters often need dynamic runtime selection and plugin-style wiring.
- **CHE-0026. Correctness-First Build Configuration**: Add CI enforcement for release-profile and lint invariants. Correctness-focused settings only protect production if every build path and crate actually uses them.
- **CHE-0027. Manual Error Trait Implementations in cherry-pit-core**: Add tests for `source()` chains and structured error contents. Error causality is critical for diagnosing distributed failures and retry decisions.
- **CHE-0028. Compile-Fail Tests as Type Safety Contracts**: Add compile-fail coverage for envelope construction and serialization-scope boundaries. The highest-risk architectural contracts are precisely the ones that prevent malformed persisted facts.
- **CHE-0029. Cargo Workspace with Layered Crate DAG**: Validate planned crate boundaries with thin spike implementations. Distributed-system abstractions often fail at integration seams, not in isolated trait design.
- **CHE-0030. Flat Public API via Private Modules**: Add a public API stability check. Event-sourcing frameworks rely on durable APIs because stored events and user code outlive internal refactors.
- **CHE-0031. MessagePack with Named Encoding for Persistence**: Specify canonical encoding and unknown-field behavior. Deterministic byte fixtures and forward compatibility depend on stable serializer settings across dependency upgrades.
- **CHE-0032. Atomic File Writes via Temp-File and Rename**: Add startup recovery for orphaned temp files and failed directory syncs. Crash consistency requires a complete recovery protocol, not just atomic rename during the happy path.
- **CHE-0033. UUID v7 for Event Identity**: Clarify monotonicity limits under clock rollback and multi-process generation. UUID v7 is sortable but not a causality source, so ordering must remain sequence-based.
- **CHE-0034. jiff::Timestamp as Temporal Foundation**: Add clock-source and testability guidance. Distributed timestamps are observational metadata, and correctness must not depend on synchronized wall clocks.
- **CHE-0035. Two-Level Concurrency Architecture**: Specify lock acquisition ordering and cancellation safety. Async mutexes and file I/O can interact badly under cancellation, causing lost progress or leaked in-memory coordination state.
- **CHE-0036. File-Per-Stream Full-Rewrite Storage Model**: Add explicit scale and migration thresholds tied to segments or snapshots. O(n) rewrites eventually threaten availability and increase crash-recovery cost for long streams.
- **CHE-0037. No Snapshot Support (Deliberate Deferral)**: Define the snapshot-introduction compatibility plan. Snapshots alter replay semantics and aggregate serialization requirements, so delaying them should still preserve a future migration path.
- **CHE-0038. Testing Strategy and Property-Based Verification**: Add fault-injection tests for crashes, duplicate delivery, and concurrency conflicts. Distributed correctness is mostly about adversarial schedules and partial failures, not only valid arbitrary inputs.
- **CHE-0039. Correlation Context Propagation**: Define correlation propagation invariants for fan-out and nested causality. Causal tracing becomes ambiguous when one event creates multiple downstream commands or a command has multiple causes.
- **CHE-0040. Saga and Compensation Patterns (Deliberate Deferral)**: Add minimum dead-letter and failed-policy-output semantics now. Even choreography needs a safe place for failed reactions or at-least-once delivery can become unbounded retry noise.
- **CHE-0041. Idempotency Strategy**: Promote command idempotency keys from sketch to first-class boundary design. Aggregate-local duplicate detection is insufficient for retried creates, external messages, and returning the original result.
- **CHE-0042. EventEnvelope Construction Invariants**: Add sequence-contiguity validation on load. Non-zero sequence numbers are not enough because replay correctness requires no gaps, duplicates, or out-of-order events per stream.
- **CHE-0043. Process-Level File Fencing**: Add stale-lock and filesystem-support guidance. Advisory locks vary across local, networked, and containerized filesystems, which affects single-writer correctness.
- **CHE-0044. Object Store Backend (Planned)** *(Proposed)*: Specify provider-specific consistency assumptions and compare-and-swap tests before adoption. Object stores differ in ETag, overwrite, listing, and read-after-write semantics, all of which affect event-store correctness.
- **CHE-0045. Serialization Scope Per Crate**: Add cross-format migration and equivalence tests. Independent serialization strategies are safe only if logical envelope semantics survive backend migration and replay across formats.

## Pardosa Domain (PAR)

- **PAR-0001. Fiber State Machine as Inspectable Data Table**: Add a compile-time or startup validator proving transition-table totality properties: no duplicate `(state, action)`, no unreachable terminal paths, and no unintended dead states. Stream processors depend on replay determinism, so the table should be tied to an invariant checker rather than tests alone.
- **PAR-0002. Index::NONE Sentinel Replacing Option\<Index\>**: Introduce a distinct `PrecursorIndex` or `OptionalIndex` wrapper so sentinel handling is localized instead of exposed through general `Index`. Sentinel encodings are efficient but dangerous in replay and migration paths; isolating nullable meaning reduces corruption risk.
- **PAR-0003. Event Immutability — Private Fields + non_exhaustive**: Add a versioned event-envelope boundary that separates immutable event identity, ordering metadata, and payload schema. Immutable fields protect local APIs, but distributed event stores need explicit envelope evolution for migration, replay, and consumer compatibility.
- **PAR-0004. Single-Writer Per Stream**: Add an explicit writer lease or leader-fencing protocol before replay/write admission, not only per-publish expected-sequence fencing. Two writers can still build divergent in-memory state before the loser’s first failed publish.
- **PAR-0005. New-Stream Migration Model**: Specify a migration state machine with durable phases, orphan cleanup ownership, and idempotent resume behavior. Copy-and-swap is sound, but crash recovery needs durable phase markers for incomplete copy, ready-to-cutover, cutover-done, and cleanup-needed states.
- **PAR-0006. Genome as Primary Serialization**: Define a formal compatibility contract for schema evolution: additive fields, removed fields, defaulting, rejected changes, and migration triggers. Schema hashes detect mismatch, but stream-processing systems need predictable upgrade rules during rollout.
- **PAR-0007. Monotonic event_id for Idempotent Publish**: Add a durable high-watermark reconciliation rule on startup that validates `event_id`, JetStream sequence, and local `Dragline` position consistency. Fault recovery should prove there are no gaps, duplicates, or regressions before accepting writes.
- **PAR-0008. Publish-then-Apply with Durable-First Semantics**: Move network publish out from under the in-memory write lock by introducing a reserved-event construction phase plus commit/apply phase guarded by fencing. Holding the state lock across network latency couples availability to broker tail latency and limits throughput under backpressure.
- **PAR-0009. LockedRescuePolicy Enum Replacing Bool**: Persist the chosen rescue policy and rescue reason as event metadata, not only as an API parameter. Audit semantics must survive replay, migration, and external inspection.
- **PAR-0010. Fallible Constructors Replacing debug_assert**: Extend validation into a single replay-time invariant gate for all persisted domain objects, not only `Fiber`. Release-safe constructors prevent new corruption, but fault-tolerant recovery must distrust durable logs.
- **PAR-0011. 64-bit Target Requirement**: Add explicit on-wire width and range invariants for all persisted numeric fields, independent of host architecture. Distributed logs outlive binaries; persisted encodings need stable numeric contracts across tooling, replay, and future platforms.
- **PAR-0012. Precursor Chain Verification on Startup**: Strengthen verification to validate per-fiber continuity against expected predecessor/current indexes, not only backward same-domain references. Stream histories can still be semantically corrupted by pointing to the wrong earlier event in the same fiber.
- **PAR-0013. NATS KV Registry for Atomic Stream Discovery**: Add a registry value schema with migration epoch, stream generation, previous stream, cutover timestamp, and writer identity. A single pointer is atomic, but consumers and operators need richer metadata to handle cutover races, replay windows, rollback, and orphan cleanup safely.
- **PAR-0014. Backpressure and Circuit Breaker**: Define explicit degraded-mode read semantics, health probes, and client-visible status rather than only rejecting writes. Serving stale reads during broker failure is a consistency contract that clients need to observe.
- **PAR-0015. Consumer Delivery Semantics and Acknowledgment Policy** *(Proposed)*: Specify idempotent consumer processing requirements and dead-letter replay/repair workflow. `AckExplicit` prevents loss, but at-least-once delivery still duplicates; exactly-once effects require consumer-side idempotency and durable checkpoints.
- **PAR-0016. Timestamp Policy and Cross-Stream Ordering** *(Proposed)*: Replace “cross-stream ordering uses JetStream sequences” with an explicit statement that there is no global cross-stream order; only per-stream order plus domain correlation keys are valid. Per-stream sequence numbers are incomparable and should not imply false causality.

## Genome Domain (GEN)

- **GEN-0001. Serde-Native Serialization with GenomeSafe Marker Trait**: Add a hard contract for how manual `Serialize`/`Deserialize` implementations are certified or rejected. Manual impls are the largest correctness hole because they can violate fixed layout, canonical encoding, and schema validation.
- **GEN-0002. No Schema Evolution — Fixed Layout**: Define the operational migration protocol that replaces schema evolution, including file discovery, dual-read windows, and failure modes. Distributed systems do not remove evolution pressure; they move it into deployment choreography.
- **GEN-0003. Compile-Time xxHash64 Schema Hashing**: Add canonical schema-source normalization rules and pinned cross-crate fixture hashes for representative types. Schema hashes are compatibility-critical; small macro or Rust syntax changes can silently fork interoperability.
- **GEN-0004. Reject Non-Deterministic Types and Serde Attributes**: Treat `#[serde(with)]` as an explicit unsafe-equivalent audit boundary with required conformance tests. Custom serde adapters can reintroduce non-determinism, omitted fields, alternate tagging, or platform-dependent encoding.
- **GEN-0005. Two-Pass Serialization Architecture**: Specify behavior for non-idempotent serialization between sizing and writing, beyond detecting size mismatch. A value that serializes twice to different same-sized payloads can pass sizing checks but violate canonical encoding.
- **GEN-0006. Zero-Copy Deserialization with forbid(unsafe_code)**: Quantify verification cost and define budgeted decode latency targets. Always-verify is the right safety default, but services need predictable tail latency under valid and adversarial inputs.
- **GEN-0007. FlatBuffers-Style Offset-Based Binary Layout**: Add a byte-level normative layout grammar for every serde data-model case. Offsets, alignment, sentinels, and heap placement are interoperability-critical; prose rules are insufficient for independent implementations.
- **GEN-0008. Transport-Agnostic Core with Companion Crate Separation**: Define the minimal transport metadata contract expected from companion crates. Bare messages rely on out-of-band content type, integrity, and routing semantics; without a contract, integrations can weaken safety assumptions.
- **GEN-0009. One Schema Per File with Embedded Schema Source**: Make the embedded schema block mandatory or define when omission is acceptable. Optional schema source harms operability and future cross-language tooling because files may lack metadata needed for inspection or migration.
- **GEN-0010. std-Only for Now — no_std Deferred**: Define criteria that would reopen `no_std`, including dependency, allocation, and API constraints. Deferred platform support should have explicit triggers to avoid ad hoc architecture changes.
- **GEN-0011. Inline Verification Check Catalog**: Convert the 20 checks into a formal validation order with fail-fast precedence. Deterministic error behavior matters for fuzzing, conformance testing, and cross-implementation compatibility.
- **GEN-0012. Little-Endian Wire Encoding — No Pointer Casts**: Add big-endian conformance tests or CI emulation requirements. The ADR claims architecture independence; byte-order correctness should be validated beyond dominant little-endian targets.
- **GEN-0013. Page-Class Resource Limits for DoS Protection**: Define service-level defaults by trust boundary rather than only page class. Safe resource budgets differ for local files, authenticated NATS, public network input, and batch replay.
- **GEN-0014. Multi-Layered Decompression Bomb Mitigation**: Require rejection or strict policy for zstd frames without content size. Optional frame metadata weakens one defense layer and complicates resource guarantees.
- **GEN-0015. Forward Compatibility Contract**: Define reserved-field semantics for future versions as an explicit compatibility state machine. Fixed version bytes are necessary but insufficient; future evolution depends on precise reader behavior for reserved bits, flags, and unknown algorithms.
- **GEN-0016. xxHash64 for File Integrity Checksums**: Add an authenticated integrity option or extension point. Non-cryptographic checksums are fine for bit rot, but distributed storage and transport often cross trust boundaries where malicious tampering must be distinguishable from corruption.
- **GEN-0017. 4 GiB Per-Message Limit — u32 Offsets**: Specify the maximum valid offset range precisely given the `u32::MAX` sentinel. Reserving `0xFFFFFFFF` means the practical maximum addressable offset is not simply all `u32` values; boundary ambiguity is a classic parser-bug source.
- **GEN-0018. Non-Zero Padding Is Hard Error**: Add canonical writer requirements for zero-initialization before every padding-emitting path. Strict reader validation only works if all writers deterministically zero padding, including future independent writers.
- **GEN-0019. Box and Arc Hash Transparency — Rc Exclusion**: Resolve the `?Sized` limitation or narrow the documented equivalence classes. Schema-hash transparency must match actual type support; otherwise compatible-looking Rust types can fail or hash inconsistently.
- **GEN-0020. Empty Containers Always Allocate Heap Entries**: Define whether identical empty heap entries may be deduplicated. Canonical encoding requires one answer; deduplicating repeated empty strings or vecs would change offsets and bytes.
- **GEN-0021. Breadth-First Heap Ordering**: Replace the informal termination argument with a precise traversal algorithm. Independent writers need deterministic queue ordering for nested structs, maps, options, and enum payloads to produce identical bytes.
- **GEN-0022. Externally Tagged Enums — Discriminant Offset Encoding**: Clarify whether enum variant addition is actually compatible under schema hashing. The layout can reject unknown discriminants, but schema hash changes may prevent old readers from reaching that check.
- **GEN-0023. i128/u128 Alignment Capped at 8 Bytes**: Add a complete alignment table for all primitive and compound types. Padding is wire-visible and correctness-critical; all implementations need the same alignment rules, not only the exceptional `i128/u128` case.
- **GEN-0024. NaN Bit-Pattern Preservation — No Canonicalization**: State that canonical encoding is bit-level, not semantic-value-level, for floats. `NaN`, `-0.0`, and `+0.0` make logical equality ambiguous; distributed deduplication must know which equality relation is authoritative.
- **GEN-0025. Bare Messages — Structural Validation Only**: Add an opt-in checksum trailer design earlier or make the transport-integrity requirement normative. Many deployments eventually use shared memory, queues, or raw sockets where “transport-protected” is not guaranteed.
- **GEN-0026. No Format Auto-Detection — Bare vs File**: Define a recommended envelope/content-type identifier for generic tooling. Explicit APIs are good, but distributed systems still need safe dispatch at boundaries that receive unknown bytes.
- **GEN-0027. Full Serde Data Model — RON Algebraic Types**: Identify which serde data-model features are supported semantically versus merely representable. “Full serde” is risky phrasing when attributes and custom serializers are constrained; interoperability depends on the exact supported subset.
- **GEN-0028. Tuple Struct / Tuple Wire Equivalence**: Strengthen the no-`decode_unchecked` invariant or remove discussion of future bypass. Type identity depends entirely on schema hash; any unchecked path would create silent nominal-type confusion.
- **GEN-0029. Reject #[serde(default)] at Compile Time**: Generalize this ADR into a policy for all no-op serde attributes. Defaults are one example of misleading serde compatibility; correctness improves if every accepted attribute has explicit binary semantics.
- **GEN-0030. Zstd-Only Compression in v1**: Specify whether compression settings affect canonical bytes. If the same logical message can be encoded with different zstd levels, canonical encoding must either exclude compression or mandate deterministic compression parameters.
- **GEN-0031. Rust-Only — Cross-Language Read Deferred**: Start the language-neutral schema export now as a read-only diagnostic artifact. Retrofitting interoperability after files exist risks incompatible hash and layout interpretations.
- **GEN-0032. Canonical Encoding Contract**: Define “same logical value” rigorously per type. Maps, floats, smart pointers, borrowed/owned strings, and wrapper transparency all create different notions of equality that affect content addressing.
- **GEN-0033. GenomeOrd Marker Trait for Map Keys**: Make custom `GenomeOrd` implementations derive-only or require a validation macro. A safe marker trait relies on user honesty; map ordering is central to canonical serialization and cross-node byte equality.
- **GEN-0034. Structured Fuzzing and Property-Based Testing for Codec Integrity** *(Proposed)*: Add coverage targets tied directly to each GEN-0011 verification check and each wire-layout type. Fuzzing is most valuable when coverage maps to specified invariants, not only arbitrary decode robustness.

## Rust Domain (RST)

- **RST-0001. Pinned Stable Toolchain with MSRV Contract**: Mechanize toolchain/MSRV consistency in CI. Reproducibility is a correctness property; drift can create different dependency resolution, lint behavior, or build artifacts.
- **RST-0002. Intentional Toolchain and Dependency Evolution**: Require dependency update PRs to include security/advisory and behavioral-risk notes. Small transitive updates can alter protocol, crypto, parsing, or async behavior with distributed impact.
- **RST-0003. Workspace Lint and Format Governance**: Add lint gates for panic/unwrap usage in service paths or boundary code. Uncontrolled panics at runtime boundaries can convert recoverable malformed input into availability failures.
- **RST-0004. Cargo Dependency Governance**: Align this ADR with SEC-0009 by adding provenance, unsafe-surface, and duplicate-version policy as one enforced dependency gate. Dependency governance and supply-chain security are the same control plane.
- **RST-0005. Workspace-Wide forbid(unsafe_code)** *(Proposed)*: Define how unsafe transitive dependency exposure is assessed and accepted. Forbidding first-party unsafe does not eliminate memory-safety risk if critical dependencies contain unaudited unsafe code.

## Security Domain (SEC)

- **SEC-0001. CISQ-Aligned Security Quality Framework**: Add explicit threat modeling as the mechanism connecting CISQ categories to concrete controls. Taxonomy prevents gaps, but threat models identify attacker capabilities and distributed trust boundaries.
- **SEC-0002. Integrity — Validate at Trust Boundaries**: Add authenticity and replay-protection requirements for accepted data, not only structural validation. Well-formed malicious or replayed messages can still violate integrity.
- **SEC-0003. Availability — Bound All Resource Consumption**: Add per-tenant/per-origin quotas and circuit breakers, not only per-input bounds. Distributed denial of service often comes from many valid bounded requests rather than one oversized request.
- **SEC-0004. Control — Restrict Capabilities by Default**: Add least-privilege runtime identity and filesystem/network capability scoping. Compile-time visibility does not constrain deployed service credentials or ambient cloud permissions.
- **SEC-0005. Authenticity — Bind Actions to Verified Origins**: Require cryptographic or platform-backed verification of origin identity before assigning correlation metadata. Correlation IDs provide traceability but not authenticity if issuers can forge them.
- **SEC-0006. Eliminate Race Conditions by Construction**: Specify fencing-token validation on every write path that depends on ownership. Single-writer claims fail under restart, delayed messages, and split brain unless stale writers are rejected.
- **SEC-0007. Confidentiality — Classify and Isolate Sensitive Data**: Mechanize secret-leak prevention with redaction types, logging tests, and serialization guards. Code review alone is weak for preventing accidental disclosure into immutable logs.
- **SEC-0008. Non-Repudiation via Append-Only Event Log**: Add tamper evidence such as hash chaining, signatures, or external anchoring. Append-only APIs do not prove stored history was not altered by privileged infrastructure access.
- **SEC-0009. Supply Chain Security via Dependency Auditing** *(Proposed)*: Promote cargo-vet or equivalent provenance review before dependency count becomes large. Known-advisory scanning misses targeted compromise and malicious-but-not-yet-reported packages.
- **SEC-0010. Transport Security via Mandatory Channel Encryption** *(Proposed)*: Add certificate lifecycle automation, expiry monitoring, and rotation testing as required controls. TLS improves confidentiality/integrity, but certificate failure can become a correlated availability failure.

## adr-fmt Domain (AFM)

- **AFM-0001. Single Source of Truth Architecture for ADR Governance**: Add explicit traceability from each invariant rule to validator code, tests, and generated guideline output. A governance control plane needs auditable consistency between specification, enforcement, and observation.
- **AFM-0002. Manual CLI Argument Parsing Over Clap** *(Superseded; stale)*: Record the precise migration compatibility expectations for users and scripts. Terminal decisions should preserve causality so future operators understand what changed at the interface boundary.
